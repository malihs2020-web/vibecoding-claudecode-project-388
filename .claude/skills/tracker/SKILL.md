# Tracker

Собирает цены на товар из нескольких источников и оформляет единую таблицу прогона (одна строка на источник). Извлечение цены с конкретной страницы не дублируется здесь — на каждом URL переиспользуется skill `extract-price` (раздел «Порядок действий»). `tracker` отвечает за то, **откуда** брать URL (фиксированный список + маркетплейсы +, при необходимости, fallback-поиск) и за сведение результатов в таблицу; `extract-price` — за то, **как** вытащить цену с уже данной страницы.

## Источники

1. **Фиксированный список URL** (см. `description` выше и раздел ниже) — проверяется всегда, браузер не требуется.
2. **Ozon / Яндекс Маркет / Wildberries** — проверяются дополнительно, если доступен браузер (эти три сайта блокируют прямые запросы, без браузера к ним не подступиться).
3. **Fallback-поиск сторонних магазинов** — вспомогательный способ добрать источники сверх фиксированного списка (например, если по товару из фиксированного списка нужно больше точек сравнения, или нужно повторить прогон для другого товара, для которого фиксированного списка ещё нет).

## Шаг 0: проверка браузера

1. Вызвать `tabs_context_mcp` (при необходимости с `createIfEmpty: true`).
2. Успешный ответ со списком вкладок → браузер **доступен** → дополнительно к фиксированному списку обработать Ozon / Яндекс Маркет / Wildberries.
3. Ошибка («Browser extension is not connected» и т.п.) → браузер **недоступен** → обрабатывать только фиксированный список URL (и, при необходимости, fallback-поиск сторонних магазинов — он тоже не требует браузера).

## Общее правило точного совпадения (для поисковых источников: Ozon, Яндекс Маркет, Wildberries, fallback-поиск)

Поиск/сортировка по цене на маркетплейсах и в поиске смешивает разные модификации товара (другой вкус, вес, размер/порода, объём упаковки). Среди найденных/отсортированных по возрастанию цены вариантов нужно оставлять для сравнения только те, чьё название/характеристики **точно** соответствуют запрошенному товару — даже если менее подходящий вариант дешевле, его нужно пропустить. Если характеристика не даёт однозначно понять соответствие по тексту (например, вес), проверять по изображению упаковки (zoom скриншота карточки) или по описанию на странице товара.

## Порядок действий: фиксированный список URL

1. Взять список URL выше (не искать и не подбирать источники самостоятельно — список фиксированный).
2. Для каждого URL по очереди вызвать skill `extract-price` (получить объект `{ regular_price, sale_price }`, либо `товар не найден` с причиной).
3. Не переопределять и не копировать логику парсинга цены внутри `tracker` — она берётся из `extract-price` как есть.

## Порядок действий: Ozon (только если доступен браузер)

Ozon блокирует прямые запросы (WebFetch/curl получают петлю редиректов через анти-бот защиту) — работать можно только через реальный браузер (Claude in Chrome).

1. Открыть ozon.ru, ввести в поиск название товара (бренд + категория + ключевые характеристики).
2. Применить доступные фильтры под запрошенные характеристики: бренд, вес (например, диапазон 10000–20000 г для «10 кг»), и т.п.
3. Включить сортировку «Дешевле» (в URL параметр `sorting=price`).
4. Считать список карточек (`get_page_text` и/или `read_page`): название, цена, наличие зачёркнутой (старой) цены.
5. Применить общее правило точного совпадения — отфильтровать карточки, не соответствующие запрошенным характеристикам.
6. Взять топ-N самых дешёвых точных совпадений. Для каждого получить прямую ссылку на карточку (`find` по названию/цене → `read_page` по `ref_id` → атрибут `href`).

## Порядок действий: Яндекс Маркет (только если доступен браузер)

Тоже требует браузера — WebFetch/curl аналогично блокируются или не отдают цену (JS-рендеринг).

1. Открыть `market.yandex.ru/search?text=<название товара>`.
2. При необходимости задать фильтры в панели слева (цена, бренд и т.п.).
3. Отдельного надёжного переключателя «сортировка по цене» на карточках маркетплейса может не быть — считывать список карточек через `get_page_text` (список подгружается частями, при необходимости повторять запрос / прокручивать страницу).
4. Применить общее правило точного совпадения.
5. Цена на карточке обычно в формате «Цена с картой Яндекс Пэй X ₽ вместо Y ₽»: `X` — это `sale_price` (цена с картой/акцией), `Y` — `regular_price`. Если указана только одна цена — это `regular_price`, `sale_price` = `null`.
6. Взять топ-N самых дешёвых точных совпадений, получить ссылку карточки (`find` → `read_page` по `ref_id` → `href`, ведёт на `/card/...`).

## Порядок действий: Wildberries (только если доступен браузер)

Тоже требует браузера.

1. Открыть `wildberries.ru/catalog/0/search.aspx?search=<название товара>`.
2. Открыть переключатель сортировки («По популярности») и выбрать «По возрастанию цены» (в URL добавляется `sort=priceup`).
3. Так как отдельного фильтра «вес» может не быть, а сортировка по цене смешивает разные объёмы упаковки (1 кг, 3 кг, 10 кг и т.д.) — задать в «Все фильтры» диапазон цены, отсекающий заведомо не подходящие по объёму варианты, либо проверять вес по каждой карточке.
4. Для карточек, где вес/вкус не виден в текстовом названии, — увеличить (zoom) скриншот карточки, чтобы прочитать вес/вкус с изображения упаковки.
5. Применить общее правило точного совпадения.
6. Взять топ-N самых дешёвых точных совпадений, получить ссылку карточки (`find` → `read_page` по `ref_id` → `href`, ведёт на `.../detail.aspx`).

## Порядок действий: fallback-поиск сторонних магазинов (вспомогательный, когда нужно расширить список)

Цель — набрать 5–10 рабочих ссылок на сторонние интернет-магазины (НЕ Ozon, НЕ Яндекс Маркет, НЕ Wildberries), по которым реально получается вытащить цену через `WebFetch`. Не требует браузера.

1. Вызвать `WebSearch` по названию/характеристикам товара, исключив домены `ozon.ru`, `market.yandex.ru`, `wildberries.ru` через `blocked_domains`.
2. Из результатов отобрать кандидатов — ссылки на конкретные карточки товара (не категории/обзоры/агрегаторы отзывов), по возможности разных магазинов.
3. Для каждого кандидата сделать пробный `WebFetch` с запросом на извлечение названия и цены.
4. Оставить только те ссылки, где `WebFetch` реально вернул читаемую цену — отсеять случаи блокировки (ошибка запроса, HTTP 4xx/498), пустого/усечённого контента и страниц, где цена подгружается через JS и не видна в ответе.
5. Применить общее правило точного совпадения к оставшимся результатам.
6. Если рабочих источников меньше 5 — при необходимости повторить `WebSearch` с уточнённым запросом. Если после разумного числа попыток рабочих источников нет вообще — сообщить об этом пользователю, не подставляя данные из сниппетов поиска вместо цены.
7. Полученные рабочие URL обработать так же, как фиксированный список (через `extract-price`).

## Оформление результата: единая таблица прогона

Собрать результаты всех источников (фиксированный список + Ozon/ЯМ/WB, если проверялись + fallback-поиск, если использовался) в одну таблицу, по одной строке на источник:

| Источник (домен) | URL | regular_price | sale_price | Статус |
|---|---|---|---|---|
| petsmart.ru | https://petsmart.ru/... | 8260.90 | null | ok |
| glavnoehvost.ru | https://glavnoehvost.ru/... | — | — | не удалось получить цену (JS, цена не отображается в ответе) |

- «Источник» — домен из URL (или «Ozon» / «Яндекс Маркет» / «Wildberries» для карточек с этих площадок).
- `regular_price` / `sale_price` — значения из объекта, который вернул `extract-price` (или прочерк, если `товар не найден`).
- «Статус» — `ok`, либо краткая причина неудачи (без выдумывания цены вместо неё). Если товар не точно совпадает по характеристикам (см. общее правило точного совпадения) — отметить это тоже, а не молча включать в сравнение.

После таблицы — коротко отметить минимальную цену среди строк со статусом `ok` и с точным совпадением товара (сравнивать `sale_price`, если он есть, иначе `regular_price`).
